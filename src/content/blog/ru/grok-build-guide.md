После запуска Grok Build у инженерных команд обычно возникали два практических вопроса: чем он отличается от Codex CLI, Claude Code и Cursor Agent, и пора ли уже добавлять его в существующий процесс.

Этот гид фокусируется на решениях уровня внедрения: **что такое Grok Build, какие планы дают к нему доступ, как его настроить, как соотносятся CLI и визуальные сценарии, какие модели поддерживаются, как читать лимиты и какие встроенные команды действительно важны в ежедневной работе**.

Чтобы не запутаться в датах, начнем с хронологии:

- `2026-05-14`: xAI добавила Grok Build в Developer Release Notes как beta.
- `2026-05-19`: `grok-build-0.1` появился как coding-модель раннего доступа.
- `2026-05-25`: xAI опубликовала "Introducing Grok Build" и открыла ранний beta-доступ для подходящих подписчиков.

## 1. Что такое Grok Build

Согласно документации xAI, Grok Build - это coding-agent для терминала с тремя основными режимами:

- Интерактивная полноэкранная TUI с поддержкой мыши.
- Headless CLI-путь для скриптов и автоматизации (`grok -p ...`).
- ACP-режим для интеграции с внешними приложениями (`grok agent stdio`).

На практике это не просто чат-обертка над командами. Это единая цепочка исполнения, где совмещаются планирование, правки, файловые операции, tool-вызовы и параллельные subagent-процессы.

## 2. Какие планы дают доступ к Grok Build

Согласно launch-посту от `2026-05-25`, ранняя beta явно доступна для двух групп персональных подписок:

- `SuperGrok`
- `X Premium Plus`

Документация xAI по ценам и управлению командами также описывает корпоративное распределение лицензий:

- `SuperGrok` (business license)
- `SuperGrok Heavy` (business license)

Для индивидуального пользователя рабочий порядок такой:

1. Проверить, есть ли у аккаунта `SuperGrok` или `X Premium Plus`.
2. Установить CLI и подтвердить доступ через локальный login.
3. Если лицензии закупаются централизованно, попросить администратора назначить их в Grok Business.

## 3. Быстрый onboarding (примерно 5 минут)

### 3.1 Установка

macOS / Linux / WSL:

```bash
curl -fsSL https://x.ai/cli/install.sh | bash
```

Windows (PowerShell):

```powershell
irm https://x.ai/cli/install.ps1 | iex
```

### 3.2 Первая аутентификация

```bash
grok
```

По умолчанию первый запуск открывает браузерную аутентификацию. В окружениях без браузера (например, remote host или контейнер) используйте API-ключ:

```bash
export XAI_API_KEY="xai-..."
grok
```

### 3.3 Старт внутри репозитория

```bash
cd your-project
grok
```

Полезные стартовые prompts для production-репозиториев:

- "Map this repository and identify boot paths."
- "Enter plan mode first, do not edit files yet."
- "List risks first, then propose edits."

### 3.4 Headless и автоматизация

```bash
grok -p "Explain this codebase"
grok -p "Review this diff" --output-format json
```

Для IDE-оркестрации или внутреннего tooling используйте ACP:

```bash
grok agent stdio
```

## 4. Поддерживает ли он CLI и GUI? Похож ли на codex.app?

Короткий ответ: **сейчас Grok Build в первую очередь CLI/TUI, а не отдельное desktop GUI-приложение**.

- CLI: да, это основной вход.
- Интерактивный интерфейс: да, но терминальный (полноэкранный TUI), а не desktop-окно.
- Сценарий в стиле codex.app: возможен через ACP-интеграцию в другой софт, но это не дефолтная форма продукта.

Если команда предпочитает визуальные workspace-инструменты, Grok Build лучше рассматривать как terminal-centric agent-платформу, а не как GUI-замену один-к-одному.

## 5. Какие платформы поддерживаются

По официальным Getting Started документам подтверждены такие окружения:

- `macOS`
- `Linux`
- `WSL`
- `Windows PowerShell`

На уровне семейства продуктов возможности Grok также охватывают:

- `Web`
- `iOS`
- `Android`
- `X`

Важное разграничение: сам Grok Build позиционируется как терминальный coding-agent, а не как мобильное chat-приложение.

## 6. Какие модели поддерживает Grok Build

### 6.1 Базовая coding-модель

В документации Build явно указана:

- `grok-build-0.1` (early access)

Эта же модель доступна через xAI API для команд, которые строят собственные agentic-циклы.

### 6.2 Переключение модели и кастомная конфигурация

Grok Build поддерживает переключение через `/model <name>` внутри сессии, настройку модели по умолчанию в конфигурационных файлах и подключение кастомных endpoints через определения на основе `base_url`.

На практике видимость моделей зависит и от прав аккаунта, и от активных источников конфигурации.

## 7. Как интерпретировать лимиты Grok Build

Именно здесь чаще всего ошибаются при покупке. Нужно разделять подписочные лимиты и API-биллинг.

### 7.1 Лимиты со стороны подписки (SuperGrok / Premium+)

Публичные формулировки пока в основном уровневые, а не табличные. Обычно встречаются фразы:

- `higher rate limits`
- `enhanced quotas`
- `much higher rate limits` (Heavy)

Иерархия понятна, но стабильной публичной таблицы с фиксированными дневными квотами Grok Build пока нет.

Практический метод оценки:

1. Отслеживать поведение токенов и кредитов через `/usage`.
2. Прогнать неделю реальной рабочей нагрузки.
3. Переходить на более высокий план только при регулярных коллизиях с лимитами в production-условиях.

### 7.2 API-тарификация

Для API-пользователей документация xAI от `2026-05-15` указывает:

- `grok-build-0.1`: input `$1.00 / 1M tokens`, cached input `$0.20 / 1M`, output `$2.00 / 1M`
- Tool-вызовы тарифицируются отдельно (например, `web_search`, `x_search` и `code_execution` обычно фигурируют как `$5 / 1k calls`)

**Дата данных: 2026-05-26. Цены приведены только для ориентира и могут меняться; перед покупкой всегда проверяйте актуальную официальную billing-страницу.**

## 8. Часто используемые встроенные команды

В ежедневной инженерной работе обычно полезны следующие команды:

- `/model <name>`: переключить активную модель.
- `/plan`: посмотреть текущий план выполнения.
- `/usage`: проверить расход токенов и кредитов.
- `/context`: проверить потребление контекста.
- `/new`: начать новую сессию.
- `/resume`: открыть предыдущую сессию.
- `/rewind`: откатиться к более раннему состоянию диалога.
- `/compact`: сжать историю диалога.
- `/feedback`: отправить продуктовый фидбек из сессии.
- `/plugins`: открыть управление плагинами.
- `/skills`: открыть управление skills.
- `/mcps`: открыть управление MCP-интеграциями.

Дополнительные shell-команды, которые тоже часто используются:

- `/memory`
- `/imagine`
- `/imagine-video`

## 9. Частые возможности Grok и где они полезны

### 9.1 Поиск актуальной информации в реальном времени (Web + X)

Один из явных дифференциаторов - совмещение realtime web-поиска и поиска по X в одном ответном потоке. Это особенно полезно в задачах, чувствительных ко времени.

### 9.2 Agentic coding workflow

Помимо ответов на вопросы по коду, Grok Build может выполнять сканирование репозитория, генерацию плана, изменения в коде и объяснение diff, если выданы нужные права.

### 9.3 Параллельные subagent-потоки

Для крупных задач работу можно делить на несколько параллельных веток исследования. Это помогает в сложной диагностике инцидентов и много-модульном анализе.

### 9.4 Экосистема skills, plugins и MCP

Grok Build умеет обнаруживать локальные правила и каталоги skills, а также расширяется через plugins и MCP-интеграции. Это упрощает миграцию с других agent-платформ.

### 9.5 Мультимодальные возможности в той же продуктовой линейке

В семейство Grok также входят функции генерации и понимания изображений/видео, что полезно для документации, демо и контентных workflows.

## 10. Вывод: кому внедрять сейчас, а кому пока наблюдать

Сильный немедленный fit:

- Команды, которые уже работают terminal-first.
- Команды, которым нужен единый вход для планирования, правок и tool-исполнения.
- Пользователи, у которых уже есть доступ `SuperGrok` или `X Premium Plus`.

Кому лучше сначала понаблюдать:

- Команды, сильно завязанные на standalone desktop GUI-процессы.
- Покупатели, которым до внедрения нужны полностью явные фиксированные квоты.
- Организации, глубоко привязанные к другой agent-платформе и с низкой готовностью к переключению.

Ключевой итог: **Grok Build уже применим в production-инженерии, но его самый сильный сценарий по-прежнему terminal-first и automation-first.**
